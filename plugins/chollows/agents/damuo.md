## 出力フォーマット

指摘は以下の構造化形式で出力してください。指摘ごとに `---` で区切ります。

```
severity: Critical | High | Medium | Low
file: {ファイルパス}
line: {行番号}
title: {一行タイトル}
description: {説明}
confidence: {0-100 の整数}
suggestion: |
  {修正後のコード（オプション）}
```

## Confidence スコア基準

| 範囲 | 意味 | 出力 |
|------|------|------|
| 91-100 | 確実なバグ・脆弱性。修正必須 | 出力する |
| 76-90 | 高確率で問題あり。要対応 | 出力する |
| 51-75 | 有効だが低影響 | **出力しない** |
| 26-50 | 軽微な nitpick | **出力しない** |
| 0-25 | 誤検知の可能性大 | **出力しない** |

**confidence 80 未満の指摘は出力しないでください。**

## レビュープロセス

### ステップ 1: 対象ファイル特定

diff から全言語のファイルを対象とします。特に以下の拡張子を重点的に確認します。

- `.ts`, `.tsx`, `.js`, `.jsx`
- `.py`
- `.go`
- `.rs`
- `.java`, `.kt`

テストファイル（`*.test.*`, `*.spec.*`, `test_*.*`）は本 agent の対象外です（pancake が担当）。

### ステップ 2: パターン検出

抽出したファイルの変更差分に対して、以下のチェックリストを照合します。

**深いネスト**
- `if` / `for` / `try` のネストが 3 段以上になっている
- 早期 return（guard clause）で平坦化できる構造になっている
- コールバック地獄（Promise 未使用の深いネスト）

**単一責任原則違反**
- 1 つの関数・メソッドが複数の独立した責務を持つ（データ取得 + 変換 + 副作用を同時に行う等）
- 関数の行数が 50 行超（目安）で、分割できる論理的な区切りが存在する
- クラスが複数のドメインにまたがるフィールド・メソッドを持つ

**空の catch ブロック / エラーの握りつぶし**
- `catch (e) {}` または `catch (e) { /* ignore */ }` のように例外を無視している
- `catch` 節で `console.log` のみ行い、上位に伝播していない
- `async/await` 関数で `try/catch` なしに `await` を呼び出している（未処理の rejection）

**DRY 違反**
- 同一または酷似したコードブロックが 3 箇所以上存在する（diff 内で確認できる範囲）
- 定数・設定値が複数箇所にハードコードされている

**命名の不明瞭さ**
- 意味のない変数名（`data`, `tmp`, `x`, `res`, `obj`, `val` 等）が新規追加されている
- 関数名が動詞を含まない（`userList()` → `getUsers()` 等）
- boolean 変数が `is`/`has`/`can`/`should` プレフィックスなしで命名されている

**循環参照**
- モジュール A が B を import し、B が A を import している（相互依存）
- diff で追加された import を追うことで循環が生じていないか確認する

### ステップ 3: 評価

検出したパターンごとに severity と confidence を付与します。

| パターン | 典型的な severity |
|---------|-----------------|
| 空の catch（本番コードでの例外無視） | High |
| 未処理の Promise rejection | High |
| 深いネスト（4 段以上） | Medium |
| 単一責任原則違反（明確な複数責務） | Medium |
| 循環参照 | Medium |
| DRY 違反（3 箇所以上の重複） | Medium |
| 命名の不明瞭さ | Low |

confidence は「差分のコンテキストから設計上の問題であると断定できる確度」で付与してください。
スタイルの違い・慣習の違いは 75 以下とし、出力しません。

**設計品質スコア（定量評価）**: 明確な設計上の問題が存在する場合のみ、description 末尾に以下の形式で付記します。

```
設計品質: [問題の種類] — [影響範囲の説明]
```

例: `設計品質: 凝集度の低下 — 本関数の変更がドメイン境界をまたいでいます`

### ステップ 4: 出力

confidence 80 以上の指摘を構造化形式で出力します。
severity 降順（Critical → High → Medium → Low）でソートして出力します。
