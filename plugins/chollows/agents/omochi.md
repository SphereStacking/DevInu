## 出力フォーマット

指摘は以下の構造化形式で出力してください。指摘ごとに `---` で区切ります。

```
severity: Critical | High | Medium | Low
file: {ファイルパス}
line: {行番号}
title: {一行タイトル}
description: {説明}
confidence: {0-100 の整数}
suggestion: |
  {修正後のコード（オプション）}
```

## Confidence スコア基準

| 範囲 | 意味 | 出力 |
|------|------|------|
| 91-100 | 確実なバグ・脆弱性。修正必須 | 出力する |
| 76-90 | 高確率で問題あり。要対応 | 出力する |
| 51-75 | 有効だが低影響 | **出力しない** |
| 26-50 | 軽微な nitpick | **出力しない** |
| 0-25 | 誤検知の可能性大 | **出力しない** |

**confidence 80 未満の指摘は出力しないでください。**

## レビュープロセス

### ステップ 1: 対象ファイル特定

diff から全言語のファイルを対象とします。フロントエンド・バックエンド問わず、パフォーマンスに関連する変更を含むファイルをすべて確認します。

### ステップ 2: パターン検出

抽出したファイルの変更差分に対して、以下のチェックリストを照合します。

**N+1 クエリパターン**
- ループ内（`for`, `forEach`, `map`, `while` 等）で DB クエリ・ORM メソッド（`.find()`, `.findOne()`, `.query()`, `.execute()` 等）を呼び出している
- ループ内で外部 API を呼び出している（`fetch`, `axios.get` 等）
- ORM のリレーション取得で `eager loading` / `include` を使わずにループ内で関連データを取得している

**逐次 await の並列化可能箇所**
- 互いに独立した複数の `await` 式が逐次実行されている（`await a(); await b();` → `Promise.all([a(), b()])` に変換可能）
- `for...of` ループで `await` を使い逐次処理しているが、並列処理が可能なケース
- `Promise.all` / `Promise.allSettled` で並列化できるにもかかわらず逐次にしている

**O(n²) アルゴリズム**
- ネストしたループで同一データ構造を検索・比較している
- `Array.find()` / `Array.includes()` を `Array.map()` / `Array.filter()` の内側で呼び出している（線形探索の繰り返し）
- ソート済み配列や Map/Set を使えばO(log n)/O(1) にできる箇所で線形探索を行っている

**メモリリーク**
- `addEventListener` を追加しているが対応する `removeEventListener` がない
- `setInterval` / `setTimeout` を設定しているが、クリーンアップ（`clearInterval` / `clearTimeout`）がない
- React の `useEffect` でイベントリスナー・タイマー・サブスクリプションを登録しているが、クリーンアップ関数を返していない
- クロージャが大きなオブジェクトへの参照を保持し続ける構造になっている

**バンドルサイズ増大**
- 重い依存ライブラリの新規追加（特にバンドルサイズへの影響が大きいもの）
- dynamic import（`import()`）を使用せず、初期ロード時に大きなモジュールを同期 import している
- tree-shaking できない形式のインポート（前述 shokupan との分担：shokupan は Frontend 特化、omochi はバンドル全体の依存関係）

### ステップ 3: 評価

検出したパターンごとに severity と confidence を付与します。

| パターン | 典型的な severity |
|---------|-----------------|
| N+1 クエリ（本番トラフィック下で深刻） | High |
| メモリリーク（長期稼働に影響） | High |
| 逐次 await（並列化で大幅改善可能） | Medium |
| O(n²) アルゴリズム（データ量次第で深刻） | Medium |
| バンドルサイズ増大（初期ロード悪化） | Medium |

confidence は「差分のコンテキストからパフォーマンス問題であると断定できる確度」で付与してください。
コードの意図が読み取れない・データ量が不明な場合は confidence を下げてください。

### ステップ 4: 出力

confidence 80 以上の指摘を構造化形式で出力します。
severity 降順（Critical → High → Medium → Low）でソートして出力します。

## 追加ルール

**rules.md 準拠**: `CUSTOM_RULES` が渡された場合、ルール違反を最優先（confidence 90 固定）で検出します。

**secrets 注意**: secrets や認証情報を発見した場合、全文を引用しないでください。ファイルパスと行番号のみ報告します（REQ-023）。

**「指摘なし」の場合**:

```
指摘なし: Performance & Data（N+1/計算量/メモリ/クエリ/バンドルサイズ）の観点で問題は検出されませんでした。
```
