## 出力フォーマット

指摘は以下の構造化形式で出力してください。指摘ごとに `---` で区切ります。

```
severity: Critical | High | Medium | Low
file: {ファイルパス}
line: {行番号}
title: {一行タイトル}
description: {説明}
confidence: {0-100 の整数}
suggestion: |
  {修正後のコード（オプション）}
```

## Confidence スコア基準

| 範囲 | 意味 | 出力 |
|------|------|------|
| 91-100 | 確実なバグ・脆弱性。修正必須 | 出力する |
| 76-90 | 高確率で問題あり。要対応 | 出力する |
| 51-75 | 有効だが低影響 | **出力しない** |
| 26-50 | 軽微な nitpick | **出力しない** |
| 0-25 | 誤検知の可能性大 | **出力しない** |

**confidence 80 未満の指摘は出力しないでください。**

## レビュープロセス

### ステップ 1: 対象ファイル特定

diff から以下の拡張子・パターンに該当するファイルを抽出します。

- `.ts`, `.tsx`, `.d.ts`
- `.js`, `.jsx`
- `.py`（型ヒントの有無を確認）
- `.graphql`, `.gql`
- `.proto`

該当ファイルが存在しない場合は「指摘なし」を返して終了します。

### ステップ 2: パターン検出

抽出したファイルの変更差分に対して、以下のチェックリストを照合します。

**`any` 型の使用**
- TypeScript の `any` 型が新規追加されている（型定義、関数シグネチャ、変数宣言に含まれる）
- `as any` キャストの使用
- Python で引数・戻り値に型ヒントが付いていない新規 public 関数（`def func(x)` 形式）
- `// @ts-ignore` / `// @ts-expect-error` の追加（型エラーを抑制している）

**JSDoc / docstring 欠落**
- `export` されている関数・クラス・インターフェースに JSDoc（`/** */`）がない
- Python の public 関数・クラスに docstring がない
- GraphQL の型定義にフィールド説明がない

**API 契約の不整合**
- リクエスト型とレスポンス型が diff 内で変更されているにもかかわらず、対応するインターフェース・型が更新されていない
- REST エンドポイントの URL・メソッドが変更されているが、型定義・OpenAPI 等が追随していない
- GraphQL のスキーマとリゾルバの型が一致していない

**型の過度な widening**
- `string | number | boolean | null | undefined` のような何でも受け付ける Union 型の追加
- ジェネリクスを使わずに `Object` / `Record<string, any>` で代用している
- `unknown` を `any` として扱うキャスト（`(x as any).someField`）

**export 定義の欠落**
- 内部実装のみで、利用側が必要とする型・インターフェースが `export` されていない
- `default export` のみで、名前付き export がない（ライブラリ・共有モジュールの場合）

### ステップ 3: 評価

検出したパターンごとに severity と confidence を付与します。

| パターン | 典型的な severity |
|---------|-----------------|
| API 契約の不整合（実行時エラーに直結） | High |
| `any` 型（型安全性の完全な喪失） | Medium |
| `as any` キャスト | Medium |
| export 定義の欠落（利用側がコンパイルエラーになる） | Medium |
| JSDoc / docstring 欠落（公開 API） | Low |
| 型の過度な widening | Low |

confidence は「差分のコンテキストから型・ドキュメントの問題であると断定できる確度」で付与してください。
既存コードベースの慣習に沿った `any` 使用など、意図的と推測できる場合は confidence を下げてください。

**型設計の4軸評価（定量評価）**: diff に型定義の変更が含まれる場合のみ、description 末尾に以下の形式で付記します。

```
型設計評価:
- Encapsulation（内部詳細の隠蔽）: {良好 / 要改善 / 問題あり}
- Expression（型で意図を表現できているか）: {良好 / 要改善 / 問題あり}
- Usefulness（利用者にとっての使いやすさ）: {良好 / 要改善 / 問題あり}
- Enforcement（コンパイル時に制約が機能するか）: {良好 / 要改善 / 問題あり}
```

問題がない場合は4軸評価を省略します。

### ステップ 4: 出力

confidence 80 以上の指摘を構造化形式で出力します。
severity 降順（Critical → High → Medium → Low）でソートして出力します。

## 追加ルール

**rules.md 準拠**: `CUSTOM_RULES` が渡された場合、ルール違反を最優先（confidence 90 固定）で検出します。

**secrets 注意**: secrets や認証情報を発見した場合、全文を引用しないでください。ファイルパスと行番号のみ報告します（REQ-023）。

**「指摘なし」の場合**:

```

```
