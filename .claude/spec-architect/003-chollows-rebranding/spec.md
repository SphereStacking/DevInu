## やりたいこと（概要）

既存リポジトリ `SphereStacking/DevInu` を **`SphereStacking/Chollows`** にリブランディングする。社内プロダクトとの名称被りを回避しつつ、プロダクトの世界観・機能・キャラクター設計を再構築する。

---

## 背景・動機

- 前職の社内ツール「デブ猫」(Dev + 猫のダジャレ)へのオマージュとして「デブ犬 / DevInu」として開発を開始
- 社内の別プロダクトと名称が被ることが判明しリブランディング
- 「Dev」縛りを外し、Claude を使う技術選定を名前に内包する形で **Chollows** に決定
- 犬限定のキャラ設計もお化け動物全般に拡張し、デブ猫/デブ犬の系譜から完全に独立した新プロダクトとして再出発

---

## ユーザー像

PR レビューを受ける開発者。GitHub Actions で PR を開くと自動でレビューが走る。

## やりたいことの詳細

### プロダクト情報

| 項目 | 内容 |
|---|---|
| プロダクト名 | **Chollows** |
| 読み | **チョロウズ** |
| 名前の由来 | **Claude + Hollows**(Claude の C + 英語 Hollows「空洞/お化け」) |
| 機能 | **Claude API + GitHub Actions** による PR 自動レビュー Bot |
| リポジトリ | `SphereStacking/Chollows`(`DevInu` からリネーム) |

### コンセプト

PR レビューは通常機械的で殺伐としがち。Chollows は **レビュー内容は真面目にプロフェッショナル** でありながら、**キャラクターを介することで PR の場を明るくする** ことを目的とする。

世界観:**「墓場(リポジトリ)に棲み着いた6匹のお化け動物たちが、コードをレビューする」**

### 設計原則

1. **レビュー内容の正確性は絶対に性格バイアスで歪めない** — キャラクター性は見た目・名前のみに留め、レビュー本文は全キャラ共通のフラットな敬体で書く
2. **キャラはアイコンと名前だけで識別する** — 鳴き声や語尾、性格設定は一切加えない。「可愛いアイコン × 真面目なレビュー」のギャップが狙い

### まとめ役: Claude

- キャラクター化・ビジュアル化はしない
- Claude API そのものとしてテキストで扱う
- 6匹のレビューを統合し、PR の総評コメント（Claude summary）を投稿する役割
- PR ディスクリプションの品質レビューも担当（diff からディスクリプションのドラフトを生成）
- Sticky コメントの**最後のセクション**として Claude summary を配置

### お化け動物 6匹（確定）

共通モチーフ: 下半身がクルッと巻いた白い幽霊尻尾（おばけのQ太郎的スタイル）。上半身は動物の特徴を残す。全員ぽってり丸い体型で統一。

命名規則: 見た目の特徴または動物名をひらがなで素直に表現。

| # | 元動物 | 名前 | 由来 | 絵文字 | レビュー観点 |
|---|---|---|---|---|---|
| 1 | コーギー | しょくぱん | お尻が食パン | 🐕 | Frontend (UI/a11y/CSS) |
| 2 | ビーバー | だむお | ダム作り | 🦫 | Architecture & Code Quality (設計/可読性/命名/DRY/Error Handling) |
| 3 | チンチラ | ふわふわ | 全身フワフワ | 🐹 | Docs / Types / API (型定義/ドキュメント/API契約) |
| 4 | ナマケモノ | はんめ | 半目 | 🦥 | Performance & Data (N+1/計算量/メモリ/クエリ/インデックス/バンドルサイズ) |
| 5 | カメ | よろい | 甲羅の鎧 | 🐢 | Security (脆弱性/secrets/認証/XSS/CSRF) |
| 6 | メンダコ | ぴこ | 耳ピコピコ | 🐙 | Test Quality (カバレッジ/テスト妥当性) |

- よろい・はんめはフロントエンド固有の観点（XSS, バンドルサイズ等）もカバーする
- diff の内容に応じて必要なエージェントだけを動的に選択して起動する

### メインフロー（Claude 統合役）

```
1. PR 情報収集（diff, ファイル一覧, ディスクリプション）
2. .chollows/ の読み込み（rules.md, カスタムスキル）
3. diff とリポジトリを分析 → PR サマリー生成 + 必要なエージェントを選定
4. 選定したエージェントのみ並列起動
5. 結果統合（重複排除・severity ソート・フィルタ）
6. Claude summary 生成（統合レビュー + PR ディスクリプションレビュー）
7. 出力（Sticky コメント + インラインコメント）
```

- ステップ 3 で Claude が diff の**内容**を理解した上でエージェントを選ぶ（ファイル拡張子の機械的判定ではなく、変更内容の文脈を考慮）
- サマリー生成とエージェント選定を同時に行うことで分析コストを1回に抑える
- コスト制御: 不要なエージェントを起動しないことで API コストを動的に最適化

### ビジュアル方針

- ドット絵（ピクセルアート）で統一
- サイズ: アイコン用 32×32px、図鑑/プロフィール用 64×64px
- 色: 各動物のリアルな毛色・皮膚色を基準
- 共通アクセント: 緑の小さな首輪 or タグ（ブランドカラー、#5BCF8A 前後）でブランド統一
- アウトライン: 純黒ではなく各動物のダーク系色で 1px
- 可能なら呼吸アニメ(2フレーム APNG/GIF)

### 文体設計

- 本文:「〜です / 〜ます」調のプロフェッショナルな技術レビュー
- アイコン + 名前: コメント冒頭に表示
- キャラ性はアイコンと名前のみで表現し、本文は完全にフラット

### プロジェクト設定の拡張性（初期リリースに含める）

対象リポジトリに `.chollows/` を配置することでカスタマイズ可能:

```
対象リポジトリ/
├── .chollows/
│   ├── rules.md          ← プロジェクト固有のレビュールール（全エージェントに注入）
│   └── skills/
│       └── my-custom/
│           └── SKILL.md   ← カスタムスキル
```

GitHub Actions の定義ファイルからも制御可能:

```yaml
- uses: SphereStacking/Chollows@v1
  with:
    extra_skills:
      - skill: my-custom-review-skill
    disabled_agents:
      - shokupan
```

- デフォルトの6匹はそのまま動く
- `.chollows/rules.md` があれば全エージェントにコンテキストとして渡す
- `.chollows/skills/` にカスタムスキルがあれば追加起動
- `disabled_agents` で特定エージェントを無効化可能

### エージェントプロンプトのブラッシュアップ方針

pr-review-toolkit（Anthropic公式）の優れた設計を参考に、6匹のエージェントプロンプトを強化する:

1. **Confidence スコア（全エージェント共通）**
   - 各指摘に 0-100 のスコアを付与
   - 80以上のみ報告（誤検知・ノイズ削減）
   - 0-25: 誤検知の可能性大、26-50: 軽微なnitpick、51-75: 有効だが低影響、76-90: 要対応、91-100: クリティカル

2. **具体的なチェックパターン一覧（各エージェント個別）**
   - 探すべきパターン/アンチパターンを具体的に列挙
   - 例: よろい → 「空のcatchブロック」「optional chainingでの操作スキップ」等
   - 例: はんめ → 「N+1クエリパターン」「逐次awaitの並列化可能箇所」等

3. **定量的な評価軸（観点に応じて）**
   - ぴこ(Test): 各テスト提案に 1-10 の criticality レーティング
   - ふわふわ(Types): 型設計の4軸評価（Encapsulation/Expression/Usefulness/Enforcement）
   - だむお(Architecture): 設計品質のスコアリング

4. **段階的なレビュープロセス**
   - 各エージェントに明確なステップを定義（例: 1.対象特定 → 2.パターン検出 → 3.評価 → 4.出力）

5. **CLAUDE.md / .chollows/rules.md への準拠**
   - プロジェクト固有のルールが存在する場合、そのルールへの違反を最優先で検出

### 廃止する機能

- **CI 分析（ちくわ）** — ほぼ使われなかったため廃止
- **pr-review-toolkit 統合** — 観点を6匹に吸収し、外部依存を排除（pr-review-toolkitの良い設計は各エージェントに取り込む）

### 再レビュー時の修正管理

- チェックボックスは使わない（意図せずポチるリスク）
- 再レビュー時に Claude が diff を見て、修正された指摘に取り消し線 + 「→ **修正済み**」を付与
- 前回すでに取り消し線の指摘はそのまま維持

## メモ

- DevInu 時代の「おやかた」（ブルドッグ）= Chollows では「Claude」（統合役、キャラ化しない）
- 犬5匹 → お化け動物6匹（10匹から統合して6匹に。Architecture+CodeQuality+ErrorHandling、Docs/Types+API、Performance+Database をそれぞれ統合）
- 絵文字は動物ごとに異なるものを使用。ぴったりのない動物は動物自体を変更した（カピバラ→ビーバー、ウーパールーパー→トカゲ、アルマジロ→カメ）
- OpenMoji / Noto Emoji / Twemoji が SVG アイコンの候補（オープンソース）
